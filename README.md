# Detecting Word-Based DGA Botnets Using BERT Model and Features of Domain Names

Reproducibility package for the paper:

> *Detecting Word-Based DGA Botnets Using BERT Model and Features of Domain Names*
> Vu Xuan Hanh, Xuan Dau Hoang, Thi Thu Trang Ninh
> Submitted to **JISIS** (Journal of Internet Services and Information Security)

This repository contains the two main training scripts that produce the four
results tables reported in the paper.

---

## 1. Overview

The proposed model is a fine-tuned **bert-base-uncased** classifier that takes
a domain name (optionally with TLD) and a vector of 11 hand-crafted lexical
features. Three training scenarios are evaluated:

| Scenario | TLD kept | + 11 features |
|:--------:|:--------:|:-------------:|
| KB1      | ✗        | ✗             |
| KB2      | ✓        | ✗             |
| KB3      | ✓        | ✓             |

Five prior word-based DGA detectors are re-implemented under the **same data
split** for a fair comparison:

1. Selvaraj & Panjanathan (2024) – RCNN-BiLSTM
2. Chen et al. (2023) – LSTM (4-input multi-branch)
3. Charan et al. (2020) – 8 lexical features + GradientBoosting (C5.0 proxy)
4. Biros & Kantor (2025) – CNN (word-level via `wordninja`)
5. Vu & Hoang (2021) – J48 / decision tree on 16 hand-crafted features

---

## 2. Data split methodology

The full 156,000-domain pool (78,000 benign Alexa + 6,000 × 13 word-based DGA
families) is split with `random_state=42`:

| Subset       | Ratio | Samples  | Used for                                                |
|:-------------|:-----:|:--------:|:--------------------------------------------------------|
| **Training** | 70%   | 109,200  | gradient descent / classifier fitting                   |
| **Validation** | 15% | 23,400   | early stopping **and** Tables 3, 4                      |
| **Testing**  | 15%   | 23,400   | final per-family Detection Rate — Tables 5, 6           |

The **testing subset contains BOTH classes**: ~11,700 DGA samples
(approximately 900 per family × 13 families) **and** ~11,700 benign samples.
Per-family DR is computed over DGA rows only; benign-side accuracy
(correctly classified as benign) is reported on the extra row labelled
`benign` at the bottom of Tables 5 and 6.

> **Note.** Tables 3 and 4 use the **validation** subset so that the test set
> remains untouched until the final per-family comparison in Tables 5 and 6.

## 3. Files

| File                 | Purpose                                                              |
|:---------------------|:---------------------------------------------------------------------|
| `trainBERTDga.py`    | Trains the proposed BERT model in 3 scenarios → Tables 3 and 5       |
| `trainBaselines.py`  | Reproduces the 5 baseline models under the same split → Tables 4 and 6 |
| `extractWord.py`     | Helper for tokenizing domain strings using `dictDGA`, `dictOnlyNLTK` |
| `requirements.txt`   | Python dependencies                                                  |

### Output files (under `results/`)

Generated by `trainBERTDga.py`:

```
results/
├── scenario_<N>_epoch_log.txt   # per-epoch metrics on train + val
├── scenario_<N>_final.txt       # Train/Val/Test + per-family DR + benign
├── summary_table3_table5.txt    # combined Table 3 (val) + Table 5 (test)
└── full_run.txt                 # console log
```

Generated by `trainBaselines.py`:

```
results/
├── baseline_<name>_epoch_log.txt
├── baseline_<name>_final.txt
├── baselines_summary_table4_table6.txt   # combined Table 4 (val) + Table 6 (test)
└── baselines_full_run.txt
```

## 4. Expected input data

Place three CSV files in `dataout/`:

```
dataout/
├── train.csv    # 70% of pool
├── val.csv      # 15% of pool — drives early stopping and Tables 3, 4
└── test.csv     # 15% of pool — sole source of Tables 5, 6
```

Each CSV must contain the following columns:

| Column   | Type    | Description                                          |
|:---------|:--------|:-----------------------------------------------------|
| `domain` | string  | full domain name (e.g. `ssfnrsensinaix.com`)         |
| `family` | string  | DGA family name (e.g. `banjori`) or `""` for benign  |
| `label`  | int     | `0` = benign, `1` = DGA                              |

The 13 supported families are:

```
banjori   bigviktor   gozi_gpl   gozi_luther   gozi_nasa   gozi_rfc4343
matsnu    ngioweb     nymaim     pizd          suppobox_1  suppobox_2  suppobox_3
```

Dictionaries (used for feature extraction) must be placed in `dictionary/`:

```
dictionary/
├── dictDGA.txt        # words observed only in DGA domains
├── dictOnlyNLTK.txt   # NLTK English words not seen in DGA
└── dictOnlyDGA.txt    # words exclusive to the DGA pool
```

## 5. Installation

```bash
# 1. Clone the repository
git clone <repository-url>
cd <repository>

# 2. Create a virtual environment (recommended)
python -m venv .venv
source .venv/bin/activate   # Linux / macOS
.venv\Scripts\activate      # Windows

# 3. Install PyTorch first, matching your CUDA version
#    Example for CUDA 12.1 (e.g. RTX 5070 Ti):
pip install torch --index-url https://download.pytorch.org/whl/cu121
#    Or, for CPU-only:
# pip install torch

# 4. Install remaining requirements
pip install -r requirements.txt
```

---

## 6. Running

### 6.1 Train the proposed BERT model (Tables 3, 5)

```bash
python trainBERTDga.py
```

This will run the three scenarios sequentially and write all summaries under
`results/`. The best checkpoint of each scenario is saved under
`models/scenario_<N>/best_model/`.

Expected runtime: ~3-5 hours for all three scenarios on an NVIDIA RTX 5070 Ti.

### 6.2 Reproduce the five baselines (Tables 4, 6)

```bash
python trainBaselines.py
```

Each baseline trains independently and writes its own
`baseline_<name>_final.txt` plus contributes to the combined summary.

## 9. Acknowledgements

The reproductions follow each original publication as faithfully as possible.
Where a paper omits architectural details, reasonable defaults are used and
marked `[best-effort]` in the model docstrings inside `trainBaselines.py`.

The official source code of the five baselines was not publicly available at
the time of this work; therefore the reproductions should be interpreted as
*independent re-implementations under a common protocol*, not verbatim re-runs
of the original artifacts.
